## CRITICAL Bugs

### BUG-001: Worker seed mismatch ‚Äî terrain inconsistency
**File:** `jardvoxel-worker.js:4`
**Severity:** Critical

The worker creates `new WorldGenerator(42)` with a hardcoded seed, but the main thread creates `new WorldGenerator(Math.floor(Math.random() * 2147483647))` with a random seed. Chunks generated by the worker will have completely different terrain than chunks generated by the main thread fallback.

**Impact:** When the Web Worker is available, chunks show wrong terrain (seed 42) mixed with correctly-generated chunks (random seed). Player sees disjointed/inconsistent world.

**Fix:** Pass the seed from the main thread to the worker via `postMessage`.

```javascript
// FIX: jardvoxel-worker.js
let world = null;
self.onmessage = (e) => {
  if (e.data.type === 'init') {
    world = new WorldGenerator(e.data.seed);
    return;
  }
  const { cx, cz } = e.data;
  const chunk = new VoxelChunk(cx, cz, world);
  chunk.generate();
  self.postMessage({ cx, cz, blocks: chunk.blocks.buffer }, [chunk.blocks.buffer]);
};
```

Also update `ChunkManager` constructor to send init message:
```javascript
this.worker.postMessage({ type: 'init', seed: world.seed });
```

### BUG-002: `_setBlockSafe` only overwrites AIR ‚Äî structures broken
**File:** `jardvoxel-engine.js:1404-1411`
**Severity:** Critical

`_setBlockSafe` only sets a block if the existing block is AIR, or if the new block is a wood type. This means structures (villages, temples, wells, mineshafts, etc.) cannot place COBBLESTONE, STONE, WATER, SANDSTONE, PLANKS, OBSIDIAN, LAVA, PRISMARINE, SNOW_BLOCK, PACKED_ICE, GRAVEL, MOSSY_COBBLE, MOSSY_STONE, TORCH, LANTERN, or any other non-wood block on top of existing terrain.

**Impact:** Structures generate incomplete ‚Äî floors have gaps, walls are partial, wells have no water, temples have missing layers.

**Fix:** Add a `force` parameter for structure placement:
```javascript
_setBlockSafe(x, y, z, block, force = false) {
  if (x >= 0 && x < CHUNK_SIZE && y >= 0 && y < CHUNK_HEIGHT && z >= 0 && z < CHUNK_SIZE) {
    if (force || this.blocks[idx] === BLOCKS.AIR || block === BLOCKS.WOOD || block === BLOCKS.BIRCH_WOOD || block === BLOCKS.SPRUCE_WOOD) {
      this.blocks[this._idx(x, y, z)] = block;
    }
  }
}
```
Then use `force = true` in all `_place*` structure methods.

### BUG-003: Survival mode `_placeBlock` doesn't check or decrement inventory
**File:** `jardvoxel.html:665-679`
**Severity:** Critical

In survival mode, `_placeBlock` places blocks without checking if the player has them in inventory. No count check, no decrement. Survival mode players have unlimited blocks.

**Impact:** Survival mode is effectively creative mode for block placement. Mining is pointless.

**Fix:**
```javascript
if (!this.creativeMode) {
  const count = this.inventory[blockType] || 0;
  if (count <= 0) return;
  this.inventory[blockType] = count - 1;
}
```

## MODERATE Bugs

### BUG-004: `blockTypeToId` missing 'mud' mapping ‚Äî swamp surfaces become air
**File:** `jardvoxel-survival-engine.js:498-511`
**Severity:** Moderate

`getSurfaceBlock` returns `'mud'` for swamp biomes (line 399), but `blockTypeToId` doesn't have a `'mud'` entry. Returns `0` (AIR). Swamp biome surfaces will be air blocks.

**Fix:** Add `'mud': 8` to the `types` mapping, or change `getSurfaceBlock` to return `'dirt'` for swamps.

### BUG-005: Duplicate faces in greedy meshing ‚Äî AIR neighbor case
**File:** `jardvoxel-engine.js:1726-1737`
**Severity:** Moderate (performance)

When the current block is AIR/WATER and the neighbor in `dir` is solid, the code creates a face with `dir` direction. This face has the wrong orientation and is a duplicate of the face created when processing the opposite direction for the solid neighbor. With `THREE.FrontSide` rendering, these are back-facing and culled, but still consume memory and processing time.

**Impact:** ~30-50% more vertices than necessary at chunk boundaries.

**Fix:** Only create cross-chunk boundary faces, skip same-chunk AIR‚Üísolid cases:
```javascript
if (block === BLOCKS.AIR || block === BLOCKS.WATER) {
  const nb = getNeighborBlock(x, y, z, dir);
  if (nb !== BLOCKS.AIR && nb !== BLOCKS.WATER) {
    const nx = x + dir[0], ny = y + dir[1], nz = z + dir[2];
    const isCrossChunk = nx < 0 || nx >= CHUNK_SIZE || nz < 0 || nz >= CHUNK_SIZE || ny < 0 || ny >= CHUNK_HEIGHT;
    if (!isCrossChunk) continue;
    // ... rest of face creation
  }
}
```

### BUG-006: `VoxelChunk.isSolid` doesn't exclude LAVA ‚Äî inconsistency
**File:** `jardvoxel-engine.js:1565-1568`
**Severity:** Moderate

`VoxelChunk.isSolid` only excludes AIR and WATER, but `ChunkManager.isSolidAt` also excludes LAVA and vegetation. Inconsistency between the two methods.

**Fix:** Align `VoxelChunk.isSolid` with `ChunkManager.isSolidAt`:
```javascript
isSolid(x, y, z) {
  const b = this.getBlock(x, y, z);
  return b !== BLOCKS.AIR && b !== BLOCKS.WATER && b !== BLOCKS.LAVA;
}
```

## MINOR Issues

### BUG-007: `Spline.evaluate` uses zero tangents ‚Äî effectively linear interpolation
**File:** `jardvoxel-survival-engine.js:130`
**Severity:** Minor

The cubic hermite formula has hardcoded `0` for both tangent values, making it smoothstep, not true cubic hermite. Comment is misleading.

**Fix:** Compute tangents from neighboring spline points, or update comment to say "smoothstep interpolation".

---

### BUG-008: Flying + ShiftLeft conflict ‚Äî Shift both speeds up and descends
**File:** `jardvoxel.html:718, 744-745`
**Severity:** Minor

When flying, `ShiftLeft` is used both as run speed modifier (line 718) and as descend key (line 745). If a player holds Shift to run faster while flying, they also descend. The two behaviors conflict.

**Fix:** Use a different key for descend (e.g. `ShiftRight`) or use `ControlLeft` for descend. Alternatively, only apply run speed when not flying.

### BUG-009: `getVegetationAt` redundant flower check
**File:** `jardvoxel-engine.js:782-793`
**Severity:** Minor (logic)

Lines 783-785 already return `FLOWER_RED` for `r < 0.10` in plains/forest/savanna. Then line 793 checks `if (biome === 'plains' && r < 0.14) return BLOCKS.FLOWER_RED` ‚Äî this is unreachable for plains because the earlier check already returned for `r < 0.12`. The `0.12 < r < 0.14` range is the only new case, but it only applies to plains.

**Fix:** Remove the redundant line 793, or restructure the vegetation logic to avoid overlapping conditions.

---

### BUG-010: Survival engine `PerlinNoise3D` has no 2D noise ‚Äî uses 3D with y=0
**File:** `jardvoxel-survival-engine.js:239-251`
**Severity:** Minor (performance)

`getContinentalness`, `getErosion`, `getWeirdness`, `getBiome` all call `fbm3D(x, 0, z, ...)` to generate 2D noise. This works but is wasteful ‚Äî the 3D noise computes a full 3D gradient when only 2D is needed. Also, `getBiome` calls `getBaseHeight` which calls `getContinentalness` + `getErosion` + `getWeirdness` again, causing redundant noise evaluations.

**Fix:** Add a `noise2D` method to `PerlinNoise3D` (or use the `PerlinNoise` class from the main engine), and cache biome results.
